# Financial Anomaly Detection Service

Сервис обнаружения финансовых аномалий в данных из регистров накопления 1С.
Позволяет аналитику вручную запускать анализ по любому источнику данных, мгновенно видеть аномалии на тепловой карте и детализировать подозрительные ячейки в виде временного ряда.

### Паттерны

- **Repository Pattern** — абстрактные интерфейсы + SQLite-реализация, легко заменяемые на PostgreSQL
- **Pydantic v2 Domain Models** — все доменные сущности описаны через `BaseModel` с полными Field-описаниями и примерами (используются в OpenAPI)
- **Typed Routes** — все endpoint'ы объявляют `request`-схему и `response_model`, FastAPI автоматически валидирует и документирует

## Типы аномалий

| Код | Описание |
|-----|----------|
| `SPIKE` | Значение отклоняется от скользящего среднего более чем на `spike_pct`% или `spike_zscore` σ |
| `TREND_BREAK` | Разворот устойчивого тренда за `trend_window` периодов |
| `ZERO_NEG` | Значение метрики ≤ 0 |
| `MISSING` | Ожидаемый период отсутствует в данных |
| `RATIO` | Отношение sum/qty выходит за пределы `ratio_min`…`ratio_max` |
| `MISSING_DATA` | Вся группа измерений отсутствует |

## Подключение источника данных

1. Откройте раздел **Источники** в UI (`/sources`) или используйте API.
2. Нажмите «Добавить источник» и заполните форму:

   | Поле | Описание | Пример |
   |------|----------|--------|
   | `id` | Уникальный идентификатор (латиница, без пробелов) | `sales_by_product` |
   | `name` | Человекочитаемое название | `Продажи по номенклатуре` |
   | `endpoint` | Базовый URL HTTP-сервиса 1С | `http://1c-server/base/hs/analytics/v1` |
   | `register_name` | Имя регистра накопления в 1С | `SalesByProduct` |
   | `dimensions` | Список измерений для группировки | `["Warehouse", "Product"]` |
   | `metric_fields.sum` | Поле суммы из строки 1С | `AmountTurnover` |
   | `metric_fields.qty` | Поле количества из строки 1С | `QuantityTurnover` |
   | `auth.type` | Тип авторизации: `none` или `basic` | `basic` |
   | `auth.user` / `auth.password` | Учётные данные для Basic Auth | — |

3. Нажмите «Проверить соединение» — сервис выполнит тестовый запрос к 1С и покажет статус и время ответа.
4. Сохраните источник. Он появится в выпадающем списке на главной странице.

## Контракт 1С HTTP

Сервис отправляет один GET-запрос на `{endpoint}/data` с тремя query-параметрами:

| Параметр | Тип | Описание |
|----------|-----|----------|
| `register_name` | `string` | Имя регистра накопления из конфигурации источника |
| `date_from` | `YYYY-MM-DD` | Начало периода (включительно) |
| `date_to` | `YYYY-MM-DD` | Конец периода (включительно) |

### Пример запроса (curl)

```bash
curl -G "http://1c-server/base/hs/analytics/v1/data" \
  --data-urlencode "register_name=SalesByProduct" \
  --data-urlencode "date_from=2026-01-01" \
  --data-urlencode "date_to=2026-03-31" \
  -u "user:password"          # только если auth.type = "basic"
```

### Пример запроса (Python)

```python
import requests

response = requests.get(
    "http://1c-server/base/hs/analytics/v1/data",
    params={
        "register_name": "SalesByProduct",
        "date_from": "2026-01-01",
        "date_to": "2026-03-31",
    },
    auth=("user", "password"),  # убрать если auth.type = "none"
    timeout=30,
)
response.raise_for_status()
rows = response.json()["data"]
```

### Ожидаемый ответ

1С **должна** возвращать JSON с полным массивом данных за период:

```json
{
  "data": [
    {
      "Период": "2026-01-31",
      "Warehouse": "Центральний",
      "Product": "Продукт A",
      "AmountTurnover": 15000.0,
      "QuantityTurnover": 30.0
    }
  ]
}
```

### Требования к ответу

- `"Период"` — обязательное поле, дата в формате `YYYY-MM-DD` или `YYYY-MM-DDTHH:MM:SS`.
- Имена полей измерений и метрик должны совпадать со значениями в `dimensions` и `metric_fields` конфигурации источника.
- 1С возвращает **весь массив за период** за один запрос — пагинация не используется.
- Если 1С недоступна или не отвечает за 10 секунд — сервис возвращает `504 Gateway Timeout`.
- Если 1С требует авторизацию — настройте `auth.type: "basic"` и укажите `auth.user` / `auth.password` в источнике данных.

### Реализация HTTP-сервиса на стороне 1С

#### Настройка в конфигураторе

1. Открыть конфигуратор → **Общие → HTTP-сервисы → Добавить**.
2. Задать **Корневой URL** сервиса: `analytics`.
3. Добавить **Шаблон URL**: `/v1/data` (метод `GET`, обработчик — функция ниже).
4. Опубликовать базу на веб-сервере (Apache / IIS) — сервис будет доступен по адресу
   `http://<сервер>/<база>/hs/analytics/v1/data`.

#### Модуль обработчика (встроенный язык 1С)

```bsl
// HTTP-сервис: ОбщийМодуль.АналитикаHTTPОбработчик (или модуль HTTP-сервиса)
// Метод GET /v1/data
// Параметры запроса: register_name, date_from, date_to

Функция ПолучитьДанные(Запрос) Экспорт

    // ── 1. Читаем параметры ───────────────────────────────────────────────
    ПараметрыЗапроса = Запрос.ПараметрыЗапроса;
    ИмяРегистра      = ПараметрыЗапроса.Получить("register_name");
    СтрДатаНачала    = ПараметрыЗапроса.Получить("date_from");
    СтрДатаОкончания = ПараметрыЗапроса.Получить("date_to");

    Если НЕ ЗначениеЗаполнено(ИмяРегистра)
        ИЛИ НЕ ЗначениеЗаполнено(СтрДатаНачала)
        ИЛИ НЕ ЗначениеЗаполнено(СтрДатаОкончания) Тогда
        Возврат ОтветОшибка(400, "Обязательные параметры: register_name, date_from, date_to");
    КонецЕсли;

    ДатаНачала    = НачалоДня(ЗначениеИзСтрокиВнутр(СтрДатаНачала));
    ДатаОкончания = КонецДня(ЗначениеИзСтрокиВнутр(СтрДатаОкончания));

    // ── 2. Проверяем, что регистр существует в метаданных ────────────────
    МетаРегистр = Метаданные.РегистрыНакопления.Найти(ИмяРегистра);
    Если МетаРегистр = Неопределено Тогда
        Возврат ОтветОшибка(404, "Регистр накопления не найден: " + ИмяРегистра);
    КонецЕсли;

    // ── 3. Динамически строим список измерений ────────────────────────────
    МассивИзмерений = Новый Массив;
    Для Каждого МетаИзмерение Из МетаРегистр.Измерения Цикл
        МассивИзмерений.Добавить(МетаИзмерение.Имя);
    КонецЦикла;

    // ── 4. Формируем текст запроса ────────────────────────────────────────
    //   Агрегируем обороты по МЕСЯЦУ; разворачивать по другому периоду —
    //   изменить "МЕСЯЦ" на "ДЕНЬ" / "КВАРТАЛ" / "ГОД"
    СписокИзмеренийВыбор  = "";
    СписокИзмеренийГруппа = "";
    Для Каждого ИмяИзм Из МассивИзмерений Цикл
        СписокИзмеренийВыбор  = СписокИзмеренийВыбор
            + "	Т." + ИмяИзм + " КАК " + ИмяИзм + "," + Символы.ПС;
        СписокИзмеренийГруппа = СписокИзмеренийГруппа
            + "	Т." + ИмяИзм + "," + Символы.ПС;
    КонецЦикла;

    // Получаем имена ресурсных полей (сумма и количество)
    МассивРесурсов = Новый Массив;
    Для Каждого МетаРесурс Из МетаРегистр.Ресурсы Цикл
        МассивРесурсов.Добавить(МетаРесурс.Имя);
    КонецЦикла;
    СписокРесурсовВыбор = "";
    Для Каждого ИмяРес Из МассивРесурсов Цикл
        СписокРесурсовВыбор = СписокРесурсовВыбор
            + "	СУММА(Т." + ИмяРес + "Оборот) КАК " + ИмяРес + "Оборот," + Символы.ПС;
    КонецЦикла;

    ТекстЗапроса =
    "ВЫБРАТЬ
    |	НАЧАЛОПЕРИОДА(Т.Период, МЕСЯЦ) КАК Период,
    |" + СписокИзмеренийВыбор +
    СписокРесурсовВыбор +
    "ИЗ
    |	РегистрНакопления." + ИмяРегистра + ".Обороты(&ДатаНачала, &ДатаОкончания, МЕСЯЦ,) КАК Т
    |СГРУППИРОВАТЬ ПО
    |	НАЧАЛОПЕРИОДА(Т.Период, МЕСЯЦ),
    |" + СписокИзмеренийГруппа +
    "УПОРЯДОЧИТЬ ПО
    |	Период ВОЗР";

    // ── 5. Выполняем запрос ───────────────────────────────────────────────
    Запрос1С = Новый Запрос(ТекстЗапроса);
    Запрос1С.УстановитьПараметр("ДатаНачала",    ДатаНачала);
    Запрос1С.УстановитьПараметр("ДатаОкончания", ДатаОкончания);
    Выборка = Запрос1С.Выполнить().Выгрузить();

    // ── 6. Сериализуем в массив объектов ─────────────────────────────────
    МассивДанных = Новый Массив;
    Для Каждого Строка Из Выборка Цикл
        Объект = Новый Структура;
        // Дата в формате YYYY-MM-DD
        Объект.Вставить("Период", Формат(Строка.Период, "ДФ=yyyy-MM-dd"));
        Для Каждого ИмяИзм Из МассивИзмерений Цикл
            Объект.Вставить(ИмяИзм, Строка[ИмяИзм]);
        КонецЦикла;
        Для Каждого ИмяРес Из МассивРесурсов Цикл
            Объект.Вставить(ИмяРес + "Оборот", Строка[ИмяРес + "Оборот"]);
        КонецЦикла;
        МассивДанных.Добавить(Объект);
    КонецЦикла;

    // ── 7. Формируем HTTP-ответ ───────────────────────────────────────────
    КорневойОбъект = Новый Структура("data", МассивДанных);
    ЗаписьJSON = Новый ЗаписьJSON;
    ЗаписьJSON.УстановитьСтроку();
    СериализаторJSON = Новый СериализаторJSON;
    СериализаторJSON.ЗаписатьJSON(ЗаписьJSON, КорневойОбъект,
        Новый ПараметрыПреобразованияJSON(ФорматДатыJSON.ISO8601));

    Ответ = Новый HTTPСервисОтвет(200);
    Ответ.УстановитьТелоИзСтроки(ЗаписьJSON.Закрыть(), "UTF-8");
    Ответ.Заголовки.Вставить("Content-Type", "application/json; charset=utf-8");
    Возврат Ответ;

КонецФункции

// Вспомогательная: возвращает JSON-ошибку с нужным HTTP-статусом
Функция ОтветОшибка(КодСтатуса, Сообщение)

    Тело = "{""error"": """ + Сообщение + """}";
    Ответ.УстановитьТелоИзСтроки(Тело, "UTF-8");
    Ответ.Заголовки.Вставить("Content-Type", "application/json; charset=utf-8");
    Возврат Ответ;
КонецФункции
```

#### Что делает обработчик

| Шаг | Действие |
|-----|----------|
| 1 | Читает `register_name`, `date_from`, `date_to` из query-строки; при отсутствии возвращает `400` |
| 2 | Проверяет существование регистра через `Метаданные.РегистрыНакопления.Найти()`; при отсутствии — `404` |
| 3–4 | Динамически строит список измерений и ресурсов по метаданным — код не требует правки при добавлении полей |
| 5 | Выполняет запрос к виртуальной таблице `.Обороты()` с агрегацией по месяцу |
| 6 | Сериализует результат: `"Период"` → строка `YYYY-MM-DD`, измерения и ресурсы — как есть |
| 7 | Возвращает `200 OK` с телом `{"data": [...]}` и заголовком `Content-Type: application/json` |

#### Проверка соединения (Test Connection)

Кнопка «Тест з'єднання» в UI и эндпоинт `POST /api/v1/sources/{id}/test` отправляют на 1С **специальный запрос без дат**:

```
GET {endpoint}/data?register_name=<имя_регистра>
```

Параметры `date_from` и `date_to` **отсутствуют** — это намеренно: цель запроса проверить доступность сервиса и существование регистра, а не получить данные.

**Как 1С должна реагировать на этот запрос:**

| Ситуация | Ожидаемый HTTP-статус | Результат в UI |
|----------|-----------------------|----------------|
| Сервис доступен, регистр существует | `200` с `{"data": []}` | ✅ `ok` |
| Отсутствуют `date_from`/`date_to` (допустимо) | `400 Bad Request` | ✅ `ok` (соединение есть) |
| Регистр не найден в метаданных | `404 Not Found` | ❌ `error` |
| Неверные учётные данные | `401 Unauthorized` | ❌ `error` |
| Внутренняя ошибка сервера | `500+` | ❌ `error` |
| Нет ответа за 10 секунд | timeout | ❌ `error` |

> Сервис считает соединение **успешным** при любом статусе кроме 401, 404 и 5xx.
> Поэтому безопасно возвращать `400` если даты не переданы — это не сломает тест.

**Рекомендуемая обработка в 1С** — добавьте в начало `ПолучитьДанные` ранний возврат для пинга:

```bsl
// Если дат нет — это запрос проверки соединения (test connection)
// Возвращаем 200 с пустым массивом вместо 500
СтрДатаНачала    = ПараметрыЗапроса.Получить("date_from");
СтрДатаОкончания = ПараметрыЗапроса.Получить("date_to");

Если НЕ ЗначениеЗаполнено(СтрДатаНачала) ИЛИ НЕ ЗначениеЗаполнено(СтрДатаОкончания) Тогда
    // Проверяем только что регистр существует
    Если Метаданные.РегистрыНакопления.Найти(ИмяРегистра) = Неопределено Тогда
        Возврат ОтветОшибка(404, "Регистр накопления не найден: " + ИмяРегистра);
    КонецЕсли;
    // Сервис жив, регистр есть — возвращаем пустой ответ
    Ответ = Новый HTTPСервисОтвет(200);
    Ответ.УстановитьТелоИзСтроки("{""data"": []}", "UTF-8");
    Ответ.Заголовки.Вставить("Content-Type", "application/json; charset=utf-8");
    Возврат Ответ;
КонецЕсли;
```

---

#### Важные замечания для разработчика 1С

- Имена полей в ответе (`Warehouse`, `AmountTurnover` и т.д.) должны **точно совпадать** с тем, что указано в `dimensions` и `metric_fields` конфигурации источника в сервисе обнаружения аномалий.
- Если в конфигурации 1С ресурс называется `Сумма`, то в `metric_fields` источника указывайте `"sum": "СуммаОборот"` (с суффиксом `Оборот`, который добавляет виртуальная таблица).
- Период агрегации (`МЕСЯЦ`) должен соответствовать ожидаемой гранулярности анализа. Для дневной аналитики замените на `ДЕНЬ`.
- HTTP-сервис рекомендуется публиковать с `Basic Auth` на уровне веб-сервера — учётные данные указываются в `auth` конфигурации источника.

Подробный контракт: `specs/001-anomaly-detection-mvp/contracts/1c-http.md`.

## Логи

Логирование реализовано через **Loguru**. Каждый HTTP-запрос фиксируется middleware:

```
2026-03-18 12:00:01 | DEBUG    | → POST /api/v1/analysis/run
2026-03-18 12:00:02 | DEBUG    | ← 200 POST /api/v1/analysis/run [843ms]
2026-03-18 12:00:05 | WARNING  | ← 504 POST /api/v1/analysis/run [10043ms]
```

Логи пишутся в два места:
- **Консоль** (stderr) — всегда, уровень DEBUG и выше.
- **Файл** `backend/logs/debug.log` — ротация при 10 МБ, хранение 7 дней.

При запуске через Docker логи файла доступны через volume `backend-logs` (см. ниже).

---

## Запуск в Docker

```bash
# Собрать образ и запустить сервис
docker-compose up --build

# В фоне
docker-compose up --build -d

# Просмотр логов контейнера
docker-compose logs -f backend
```

Сервис будет доступен на `http://localhost:8000`.

Данные и логи хранятся в именованных volumes:

| Volume | Путь внутри контейнера | Содержимое |
|--------|------------------------|------------|
| `backend-data` | `/app/data` | SQLite БД (`anomaly_detection.db`) |
| `backend-logs` | `/app/logs` | Файл логов (`debug.log`) |

Чтобы прочитать лог-файл напрямую:
```bash
docker-compose exec backend cat /app/logs/debug.log
```
