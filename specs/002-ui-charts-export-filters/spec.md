## User Scenarios & Testing *(mandatory)*

### User Story 1 — Интерактивная тепловая карта с легендой (Priority: P1)

Аналитик открывает страницу результатов анализа. Тепловая карта отображается с интерактивными ячейками: при наведении показывается всплывающая подсказка с типом аномалии, значением и отклонением. Рядом с картой расположена легенда, объясняющая цвет каждого типа аномалии. Вся карта умещается на экране без горизонтальной прокрутки.

**Why this priority**: Тепловая карта — главный инструмент обнаружения аномалий. Без понятной легенды и подсказок аналитик не может быстро интерпретировать результаты.

**Independent Test**: Можно протестировать, запустив анализ и открыв страницу `/heatmap` — карта отображается с легендой и тултипами без дополнительных действий.

**Acceptance Scenarios**:

1. **Given** результаты анализа загружены, **When** аналитик открывает тепловую карту, **Then** ячейки окрашены по типу аномалии, а легенда перечисляет все типы с цветом и описанием.
2. **Given** тепловая карта отображается, **When** аналитик наводит курсор на ячейку с аномалией, **Then** появляется подсказка с типом аномалии, текущим значением, предыдущим значением и процентом отклонения.
3. **Given** большое количество измерений (>20 строк), **When** страница загружается, **Then** карта и легенда умещаются в область экрана без горизонтальной прокрутки.
4. **Given** аномалий нет, **When** карта отображается, **Then** показывается пустое состояние с поясняющим сообщением.

---

### User Story 2 — Интерактивный временной ряд на drilldown (Priority: P2)

При клике на ячейку тепловой карты открывается страница детализации с графиком временного ряда. График интерактивен: можно масштабировать и прокручивать. Точки с аномалиями визуально выделены и снабжены тултипом. Заголовок страницы показывает измерение, по которому открыт drilldown.

**Why this priority**: Drilldown — второй ключевой экран для анализа. Интерактивность позволяет изучить паттерн во времени.

**Independent Test**: Можно протестировать независимо, открыв `/drilldown?run_id=...&dimensions=...` напрямую — график строится и является интерактивным.

**Acceptance Scenarios**:

1. **Given** аналитик кликнул на ячейку, **When** открывается drilldown, **Then** график отображает временной ряд с отмеченными точками аномалий в другом цвете/маркере.
2. **Given** график отображается, **When** аналитик наводит на точку аномалии, **Then** тултип показывает период, значение, тип аномалии и отклонение.
3. **Given** график отображается, **When** аналитик прокручивает или масштабирует, **Then** ось X/Y перемасштабируется интерактивно без перезагрузки страницы.
4. **Given** данных для временного ряда нет, **When** открывается drilldown, **Then** показывается понятное сообщение об отсутствии данных.

### User Story 3 — Экспорт результатов анализа в Excel (Priority: P2)

На странице таблицы аномалий доступна кнопка «Экспорт в Excel». При нажатии браузер скачивает файл `.xlsx` с данными текущей выборки (с учётом активных фильтров). Файл содержит читаемые заголовки на русском языке.

**Why this priority**: Аналитики работают в Excel для дальнейшего анализа и отчётности. Экспорт снимает ручное копирование.

**Independent Test**: Можно протестировать независимо на странице таблицы аномалий — кнопка экспорта генерирует корректный `.xlsx` без обращения к другим страницам.

**Acceptance Scenarios**:

1. **Given** таблица аномалий отображается (возможно, с фильтрами), **When** аналитик нажимает «Экспорт в Excel», **Then** браузер скачивает `.xlsx`-файл с теми же строками, что видны в таблице.
2. **Given** применён фильтр по типу аномалии `SPIKE`, **When** выполняется экспорт, **Then** в файле содержатся только строки типа `SPIKE`.
3. **Given** екcпортований файл відкрито, **When** аналітик переглядає колонки, **Then** заголовки читабельні українською: Період, Тип аномалії, Поточне значення, Попереднє значення, Відхилення %, Z-score, Виміри.
4. **Given** таблица пуста (нет аномалий по фильтру), **When** нажимается экспорт, **Then** скачивается файл только с заголовками без строк данных.

### User Story 4 — Гибкая фильтрация и поиск аномалий (Priority: P3)

На странице таблицы аномалий расширяется панель фильтров: мультивыбор типов аномалий, текстовый поиск по значениям измерений, фильтр по минимальному проценту отклонения. Фильтры применяются сразу при изменении.

**Why this priority**: Гибкая фильтрация нужна при большом количестве аномалий. Текущие фильтры уже работают — это расширение существующей функциональности.

**Independent Test**: Можно протестировать независимо на странице `/table` — добавить различные фильтры и убедиться, что таблица обновляется корректно.

**Acceptance Scenarios**:

1. **Given** таблица содержит несколько типов аномалий, **When** аналитик выбирает типы `SPIKE` и `ZERO_NEG`, **Then** таблица показывает только аномалии выбранных типов.
2. **Given** аналитик вводит часть имени в поле поиска по измерению, **When** введено 3+ символа, **Then** таблица фильтруется по строкам, содержащим введённый текст в любом значении измерения.
3. **Given** аналитик указывает минимальный порог отклонения 50%, **When** фильтр применён, **Then** отображаются только аномалии с `|pct_change| >= 50`.
4. **Given** фильтры установлены, **When** аналитик нажимает «Сбросить», **Then** все фильтры очищаются и таблица показывает все аномалии.

---

### Edge Cases

- При экспорте выборки >5000 строк должна отображаться индикация загрузки.
- Аномалии без `pct_change` (тип `MISSING`) при фильтре по отклонению не исключаются — поле остаётся пустым.
- Измерения с кириллицей и специальными символами корректно отображаются в тултипах и экспорте.
- При ширине окна <1280px легенда и панель фильтров схлапываются или переносятся без потери функциональности.
- При открытии drilldown без `run_id` в URL показывается понятная ошибка.

## Requirements *(mandatory)*

### Functional Requirements

**Визуализация**

- **FR-001**: Система ДОЛЖНА заменить Chart.js на Plotly.js для отрисовки временного ряда на странице drilldown.
- **FR-002**: График временного ряда ДОЛЖЕН поддерживать интерактивное масштабирование (zoom) и сброс вида.
- **FR-003**: При наведении на точку данных ДОЛЖЕН появляться тултип с полями: период, значение, тип аномалии (если есть), отклонение.
- **FR-004**: Точки с аномалиями ДОЛЖНЫ визуально отличаться от нормальных точек по цвету и/или форме маркера в соответствии с типом аномалии.

**Легенда**

- **FR-005**: На странице тепловой карты ДОЛЖНА присутствовать постоянная легенда с цветом и кратким описанием каждого типа аномалии.
- **FR-006**: На графике временного ряда ДОЛЖНА присутствовать легенда, объясняющая цветовое кодирование точек (норма / тип аномалии).
- **FR-007**: Легенда ДОЛЖНА быть видна без прокрутки при разрешении экрана ≥1280×768.

**Адаптивность**

- **FR-008**: Все страницы (heatmap, drilldown, table) ДОЛЖНЫ отображаться без горизонтальной прокрутки при ширине ≥1280px.
- **FR-009**: Тепловая карта с большим числом измерений ДОЛЖНА использовать вертикальную прокрутку внутри контейнера, не нарушая layout страницы.

**Экспорт в Excel**

- **FR-010**: Страница таблицы аномалий ДОЛЖНА содержать кнопку «Экспорт в Excel», генерирующую `.xlsx`-файл на стороне клиента без обращения к серверу.
- **FR-011**: Экспортируемый файл ДОЛЖЕН содержать только строки, прошедшие текущие активные фильтры.
- **FR-012**: Экспортируемый файл ДОЛЖЕН содержать колонки: Период, Тип аномалии, Измерения, Текущее значение, Предыдущее значение, Отклонение %, Z-score.
- **FR-013**: Заголовки колонок в Excel ДОЛЖНЫ бути на українській мові.

**Фильтрация**

- **FR-014**: Панель фильтров ДОЛЖНА поддерживать мультивыбор типов аномалий.
- **FR-015**: Панель фильтров ДОЛЖНА содержать текстовое поле поиска по значениям измерений (подстрока, без учёта регистра).
- **FR-016**: Панель фильтров ДОЛЖНА содержать числовое поле «Минимальное отклонение %» для фильтрации по `|pct_change|`.
- **FR-017**: Все фильтры ДОЛЖНЫ применяться немедленно при изменении (debounce 300ms для текстовых полей).
- **FR-018**: Кнопка «Сбросить фильтры» ДОЛЖНА очищать все активные фильтры.

**Строга типізація (OpenAPI → Pydantic)**

- **FR-021**: API-маршрути `/heatmap` і `/timeseries` ПОВИННІ використовувати Pydantic v2 `response_model` замість неструктурованого `dict`, щоб OpenAPI-специфікація точно описувала схему відповіді.
- **FR-022**: Клітинки теплової карти в API-відповіді ПОВИННІ містити поля `pct_change`, `current_value`, `previous_value` для відображення в тултипах (підсказках при наведенні).
- **FR-023**: Назви полів у JavaScript-коді шаблонів ПОВИННІ точно відповідати іменам полів Pydantic-схем, що публікуються через OpenAPI (перевірка по `/openapi.json`).

**Стили**

- **FR-019**: Візуальне оформлення ПОВИННО відповідати поточним стилям проекту без введення нових CSS-фреймворків.
- **FR-020**: Усі нові JS-бібліотеки ПОВИННІ підключатися через CDN без інструменту збірки.

### Key Entities

- **AnomalyExportRow**: Плоская запись для экспорта — период, тип, все пары измерение:значение, числовые метрики.
- **FilterState**: Активный набор фильтров — массив типов, текстовый запрос, минимальное отклонение, диапазон дат.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Аналитик понимает тип аномалии на тепловой карте без обращения к документации — легенда видна на одном экране.
- **SC-002**: Тултип на точке графика или ячейке карты появляется менее чем через 100мс.
- **SC-003**: Экспорт выборки из 1000 аномалий в Excel занимает менее 3 секунд.
- **SC-004**: Применение любого фильтра обновляет таблицу менее чем за 500мс.
- **SC-005**: Все страницы корректно отображаются при ширине окна от 1280px — горизонтальная прокрутка отсутствует.
- **SC-006**: Все JS-библиотеки подключаются через CDN без изменения сборочного пайплайна.
- **SC-007**: Существующие 57 backend-тестов продолжают проходить без изменений.
